    • Basics

    Just like URL's of the world wide web, linkspace looks a lot like a filesystem.
    To understand the difference, think about what happens when we merge two instances. 

    +
|-image
| |-Broken.jpg : {imgdata}
|
|-thread
  |-Coffee machine broke!
    |- msg     : {fix pls? image/Broken.jpg}
    +
    +
|-thread
  |-Tabs or spaces?
    |- msg     : {Why not U+3164?}
    =
    +

|-image
| |-Broken.jpg : {imgdata}
|
|-thread
  |-Tabs or spaces?
  | |- msg       : {Why not U+3164?}         
  |
  |-Coffee machine broke!
    |- msg       : {fix pls? image/Broken.jpg}

    A web-servers is software that serve such a file/folder tree and does some additional processing1. When a user wants to add new entries they might overlap with previous stuff. A server would have multiple options. After all, it is the only 'real' copy we can point to. However, this has profound consequences.

    In linkspace two entries can always overlap

    +
|-thread
  |-Tabs or spaces?
    |- msg : [HASH_0]{Why not U+3164?}
    +
    +
|-thread
  |-Tabs or spaces?
    |- msg : [HASH_1]{Get a life}
    =
    +
|-thread
  |-Tabs or spaces?
    |- msg : [HASH_0]{Why not U+3164?}
             [HASH_1]{Get a life}

    The hash is a unique number for every entry. It depends in part on the content and the path.
    This allows us to merge two trees.
    What is more, we also get the option to pick if we want to refer to things by a path "image/Broken.jpg" or by their hash. 

    +
|-image
| |-Broken.jpg : [HASH_2]{imgdata}
|
|-thread
  |-Coffee machine broke!
    |- msg     : [HASH_3]{fix pls? Look at [HASH_2] }
    +
    +
|-thread
  |-Coffee machine broke!
    |- msg     : [HASH_4]{
                  Hey [HASH_3]!
                  Isn't this [HASH_2] image from 2015?
                 }
    =
    +
|-image
| |-Broken.jpg : [HASH_2]{imgdata}
|
|-thread
  |-Coffee machine broke!
    |- msg     : [HASH_3]{fix pls? Look at [HASH_2] }
               : [HASH_4]{
                  Hey [HASH_3]!
                  Isn't this [HASH_2] image from 2015?
                 }

    Linkspace goes further. Each entry is set inside a Domain and a Group. A domain name is chosen by a developer creating a (graphical) interface. The domain name signals what app to use, and how things are organized. The group is set by the user. It exits because we say it exists. Finally, each entry has a creation date and can be signed. 

    +domain=message-board
+group=[#:example:com]        
|-image
| |-Broken.jpg : [HASH_2][1/13/2015]{imgdata}
|
|-thread
  |-Coffee machine broke!
    |- msg     : [HASH_3][3/3/2023]
                 {fix pls? Look at [HASH_2]}
                 [@:alice:accounting:example:com]
               : [HASH_4][4/3/2023]
                 {
                  Hey [HASH_3]!
                  Isn't this [HASH_2] image from 2015?
                 }
                 [@:bob:maintenance:example:com]

    These are the basics ideas2..
    For more details and some advanced concepts see the Guide.

    1. Databases aren't magic. They create multiple sorted lists under table names. In essence : 'table_name/primary_key=value' The tree described here is the general case where primary_key has variable length and tables can be nested and (unambiguously) merged. ↩︎
